## About this file (sidecar protocol)

This file is the **sidecar** for M-daily-scan. It records hinge-log
entries for the daily-scan cadence alongside the mission's primary
artefacts (probes, briefs, workups, frames).

**Why a sidecar for a daily-scan mission?** M-trip-journal's completion
criterion 7 requires adoption by a second mission. M-daily-scan is that
second mission. Adopting the hinge-log apparatus here tests whether
the method generalises from slow mission-theoretical work to fast
landscape-intelligence work. This is the first applied-chaos-theory
perturbation from a different **use-mode**, not just a different hinge
within the same mode.

**Cadence.** Hinge-based, not daily. A daily scan produces a brief;
a **hinge-log entry** appears only when the scan crosses a natural
cleavage — a probe-design shift, a surprising find, a method-failure,
a capability-mapping move. Fill the five slots only where load-bearing
(per `structure/hinge-entry-slots.flexiarg`).

**Voice — I-1 discipline.** Per M-trip-journal sidecar: operator (Joe)
writes first-person, agent writes as agent with attribution.

**What goes in.** Hinge-log entries from daily scans. Cross-day
structural observations that don't fit in any single brief. Probe-
design decisions that emerged from use. Applied-chaos-theory events
(method perturbations absorbed or refused).

**What does NOT go in.** The scan briefs themselves (those live in
`data/briefs/`). Cross-day aggregation. Summarised entries.

**Learning-loop channels** (per M-trip-journal): pattern library,
operator vocabulary, connection-graph, question pool. Each hinge
entry records which channels fired.

### Hinge: scoring-blindspot-reconfirmed

**Date:** 2026-04-18
**Phase context:** After the second scan run, the Orcro-orbit probes'
highest-scoring items (Orcro/urltester at −2.0, eclipse-corinthian/
Precedent_Docs at 2.0) reconfirm the scoring blind spot that Day 3

**[agent, claude-11]** — Day 3's action #5 (scoring function fix for
pre-commercial finds) now has two independent data points. This
crosses a threshold from "one-off observation" to "structural
feature of the scan infrastructure."

**Q→A:**
Q: Is the scoring function failure a bug, or a feature exposed by
use?
A: A feature. The scoring function was designed for the M-f7-lead-
   report landscape-scan use-case (find commercial repos from a
   cold landscape). It penalises the exact kind of repo that a
   warm-contact direct-target probe surfaces: small, pre-commercial,
   often license-less, personal or small-org work. When the probe
   is direct-target, the score should be near-zero-weighted and
   rank by recency/activity instead. Day 3 already named this;
   Day 4 confirms it applies generally, not just to AIQA.

**Fields:**
- Dominant tension (saññā): General-purpose scoring vs direct-target
  ranking. The scoring function is correctly pessimistic for
  landscape scans and incorrectly pessimistic for audience scans.
- Shift (saṅkhāra): Reclassifying the scoring-function issue from
  "bug in one case" to "mode-mismatch: scoring function is
  landscape-tuned, needs a per-probe-mode switch."
- Residual unknowns: Whether the fix is "direct-target bypass" (flat
  weights for `user:` and `org:` probes) or a more general per-probe
  weighting scheme.

**Learning-loop update:**
- Channel: pattern library — candidate new pattern
  `structure/signal-mode-mismatch` (or similar): when a scoring
  function is used in a mode different from its design mode,
  scores drift from calibrated to noise. Holds until a second
  application surfaces. Not written today.
- Channel: connection-graph — the scoring-blindspot observation
  now has two known instantiations (AIQA, Orcro). This turns a
  memory-footnote into a structural fact.
- Channel: question pool — possibly a future pattern "when is a
  low score actually a high-signal find?" — deferred until the
  shape firms up.

**Outcome:** Workup section 4 (infrastructure findings) will note
